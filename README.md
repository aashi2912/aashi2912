## 💡 How I think about product

<br/>

<div align="center">

*"The best AI products don't feel like AI. They feel like someone finally understood the problem."*

</div>

<br/>

```
Ship small, learn fast, kill what's not working — and have the courage to say so.
```

<br/>

| I believe | I don't believe |
|---|---|
| 🎯 The user's real problem > the stated requirement | 🚫 AI is a solution looking for a problem |
| 📐 A good PRD saves 10x the engineering time | 🚫 Moving fast means skipping clarity |
| 🔍 Talking to users is never optional | 🚫 Internal proxies understand end users |
| 🛡️ Security and compliance aren't afterthoughts | 🚫 You can bolt guardrails on later |
| 📊 If you can't measure it, you didn't ship it | 🚫 Adoption is someone else's problem |

<br/>

> I don't believe in building features nobody asked for.
> For me, **AI isn't the product — it's the tool**.
> The product is the problem you're solving, and how seamlessly you solve it.
> If the user has to think about the AI, you've already lost.
